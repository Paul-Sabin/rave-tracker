  <action>
    Present the user with three hosting provider options and wait for their selection. Research (12-RESEARCH.md) evaluated three PaaS options. All support Python ASGI, managed PostgreSQL, automatic SSL, custom domains, and git-push deployment.

    This application requires:
    - Two separate processes: web (gunicorn + uvicorn) and scheduler (APScheduler)
    - Managed PostgreSQL with automated backups (HOST-02 requirement)
    - Automatic SSL via Let's Encrypt (HOST-03)
    - Git-push deployment (HOST-05)

    The Procfile and .env.example already exist from Phase 11.

    Options:

    **Render (Recommended):**
    Pros: Native automated PostgreSQL backups (no extra config). Native background worker service type (perfect for scheduler). Flat monthly pricing ($7 web + $7 worker + $7 starter DB = ~$21/mo). Simple render.yaml for multi-service config. Health check path built into service config.
    Cons: Slightly higher base cost than Railway for low-traffic apps. Free tier spins down after inactivity (paid plan needed for scheduler).

    **Fly.io:**
    Pros: Process groups in fly.toml (web + scheduler in one config). Managed Postgres with backups. Global distribution if needed later. Usage-based pricing.
    Cons: Higher PostgreSQL cost ($38+/mo for managed basic). CLI-first workflow (flyctl required). More complex networking (IPv6 required for SSL). Steeper learning curve.

    **Railway:**
    Pros: Usage-based pricing (potentially cheapest for low traffic). Fast deploys with Railpack. Simple UI. Multiple services from same repo.
    Cons: No native PostgreSQL backups (requires third-party template). Backup blind spot is a risk for data loss. Slightly more manual multi-service setup.

    Wait for user to select: render, fly, or railway.
  </action>

    ```yaml
    databases:
      - name: rave-tracker-db
        databaseName: rave_tracker
        plan: starter
    ```

    Notes on render.yaml:
    - `sync: false` means "secret -- set manually in Render dashboard, do not sync from file"
    - `fromDatabase` auto-injects the DATABASE_URL from the managed PostgreSQL instance
    - `healthCheckPath: /health` uses the endpoint created in Phase 11
    - WEB_CONCURRENCY=2 is appropriate for Render's starter instances (avoid memory pressure)
    - Scheduler worker gets same env vars but no PORT (it's not a web service)
    - Using `runtime: docker` instead of `runtime: python` for explicit control over build steps (psycopg2 needs libpq-dev)

    **Dockerfile** -- Production container image:
    ```dockerfile
    FROM python:3.11-slim

    # Install PostgreSQL client libraries (required for psycopg2 source build)
    RUN apt-get update && apt-get install -y --no-install-recommends \
        libpq-dev gcc \
        && rm -rf /var/lib/apt/lists/*

    WORKDIR /app

    # Copy requirements first for Docker layer caching
    COPY requirements.txt .
    RUN pip install --no-cache-dir -r requirements.txt

    # Copy application code
    COPY . .

    # Default command (overridden by render.yaml / Procfile per service)
    CMD ["gunicorn", "ra_tracker.web.app:app", "--bind", "0.0.0.0:8080", "--workers", "2", "--worker-class", "uvicorn.workers.UvicornWorker", "--graceful-timeout", "30", "--timeout", "60"]
    ```

    Notes on Dockerfile:
    - python:3.11-slim for small image size
    - libpq-dev + gcc required for psycopg2 source compilation (NOT psycopg2-binary per Phase 11 decision)
    - Layer caching: requirements.txt copied before app code so pip install is cached when only code changes
    - Default CMD matches Procfile web entry, but render.yaml overrides per service
    - The scheduler service uses: `python -m ra_tracker.main --scheduler-only -c config.yaml`

    **For Render specifically:** render.yaml startCommand overrides Dockerfile CMD:
    - Web service: Uses Dockerfile CMD (gunicorn) by default, or add explicit startCommand
    - Scheduler: Needs explicit startCommand in render.yaml: `python -m ra_tracker.main --scheduler-only -c config.yaml`

    Update render.yaml to add explicit startCommands for both services for clarity.
