Create `ra-tracker/ra_tracker/observability/access_log_middleware.py`:

1. Import `time`, `logging`, and Starlette types (`Request`, `Response` from starlette, or use FastAPI's `Request`).
2. Get a logger: `logger = logging.getLogger("ra_tracker.access")` — this inherits from the `ra_tracker` logger configured in logging_config.py, so it gets the JSON formatter, correlation_id filter, and Better Stack handler automatically.
3. Create a Starlette `BaseHTTPMiddleware` subclass called `AccessLogMiddleware`:
   - In `dispatch(self, request: Request, call_next)`:
     - Record `start = time.perf_counter()`
     - Call `response = await call_next(request)`
     - Calculate `duration_ms = round((time.perf_counter() - start) * 1000, 1)`
     - Log at INFO level using `logger.info()` with `extra={}` containing structured fields:
       - `method`: `request.method` (GET, POST, etc.)
       - `path`: `request.url.path` (just the path, not full URL)
       - `status_code`: `response.status_code` (integer)
       - `duration_ms`: calculated duration
       - `query`: `str(request.query_params)` if non-empty, omit if empty
     - The message itself should be a human-readable summary like `"{method} {path} {status_code} {duration_ms}ms"` — the extra fields are what matter for structured search.
     - Return `response`
   - Skip logging for `/health` endpoint to avoid noise from load balancer health checks (check `request.url.path == "/health"`).
   - Skip logging for static file requests (`request.url.path.startswith("/static")`) to reduce noise.

Then modify `ra-tracker/ra_tracker/web/app.py`:

1. Add import: `from ..observability.access_log_middleware import AccessLogMiddleware`
2. Register the middleware BEFORE the CSRF middleware line (i.e., add `app.add_middleware(AccessLogMiddleware)` as the FIRST `add_middleware` call in `create_app()`). FastAPI processes middleware in reverse order of addition, so adding it first means it runs last (outermost to innermost: CorrelationIdMiddleware -> CSRFMiddleware -> AccessLogMiddleware). This is correct because:
   - CorrelationIdMiddleware generates the request_id FIRST (outermost)
   - AccessLogMiddleware runs LAST, so it captures the final status_code after all other middleware has processed

The middleware ordering in code should be:
```python
app.add_middleware(AccessLogMiddleware)   # Added first = runs last (innermost)
app.add_middleware(CSRFMiddleware)        # Runs middle
# CorrelationIdMiddleware added last = runs first (outermost)
```

Do NOT change the uvicorn.access log level — keep it at WARNING. The new middleware replaces the need for uvicorn access logs entirely with cleaner structured output.

The `extra` dict fields (method, path, status_code, duration_ms) are automatically included as top-level JSON fields by python-json-logger's JsonFormatter, alongside timestamp, level, logger, message, and request_id (from CorrelationIdFilter). This produces log lines like:
```json
{"timestamp": "2026-02-20T12:00:00", "level": "INFO", "logger": "ra_tracker.access", "message": "GET /rules 200 42.1ms", "request_id": "abc123def456", "method": "GET", "path": "/rules", "status_code": 200, "duration_ms": 42.1}
```
