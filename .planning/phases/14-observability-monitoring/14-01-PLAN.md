    **4. Create `ra-tracker/ra_tracker/observability/logging_config.py`:**
    - Define `LOGGING_CONFIG` dict using logging.config.dictConfig format
    - Filters: `correlation_id` filter using `asgi_correlation_id.CorrelationIdFilter` (uuid_length=32, default_value='-')
    - Formatters:
      - `json`: Use `pythonjsonlogger.json.JsonFormatter` (note: python-json-logger v3+ uses `pythonjsonlogger.json` not `pythonjsonlogger.jsonlogger`). Format string: `%(asctime)s %(levelname)s %(name)s %(message)s %(correlation_id)s`. Use `rename_fields` to map: asctime->timestamp, levelname->level, name->logger, correlation_id->request_id
      - `console`: Plain text format for dev: `%(asctime)s - %(name)s - %(levelname)s - [%(correlation_id)s] %(message)s`
    - Handlers:
      - `stdout`: StreamHandler to sys.stdout, uses `json` formatter + `correlation_id` filter
      - `logtail`: Use `logging.handlers.QueueHandler` wrapping a `logtail.LogtailHandler` via QueueListener (non-blocking per research pitfall 4). Only added if logtail_token is provided.
    - Loggers:
      - `ra_tracker`: INFO level, propagate=False
      - `uvicorn.access`: WARNING level (reduce noise), propagate=False
      - `uvicorn.error`: INFO level, propagate=False
    - Root logger: INFO level

    Create a `setup_logging(environment: str, log_level: str, logtail_token: str = "")` function that:
    1. Builds the LOGGING_CONFIG dict
    2. If environment == "development", uses `console` formatter for stdout handler
    3. If logtail_token is provided, sets up QueueHandler+QueueListener with LogtailHandler (use `queue.Queue(-1)` for unlimited size)
    4. Calls `logging.config.dictConfig(config)`
    5. Registers `atexit.register(queue_listener.stop)` if logtail is configured
    6. Suppresses noisy loggers: httpx, httpcore, apscheduler set to WARNING
    7. Returns the queue_listener (or None) so it can be stopped on shutdown

        - `environment=environment`
        - `traces_sample_rate=0.1` (10% per research recommendation)
        - `send_default_pii=True` (for user context)
        - `enable_logs=False` (logs go to Better Stack, not Sentry)

    - `clear_sentry_user()` helper:
      - Calls `sentry_sdk.set_user(None)`

    **IMPORTANT:** Do NOT use structlog (context variable isolation issues per research). Stick with python-json-logger + stdlib logging.

    ```python

        response = await call_next(request)

        return response
    ```

    Also, in the existing auth dependency (`auth.py` `get_current_user` or `require_auth`), add a call to `set_sentry_user(user.id, user.email)` when user is found.     **Best approach:** In `auth.py` `get_current_user()`, after successfully finding a user, call `set_sentry_user(user.id, user.email)`. Import is simple and direct. Then in the middleware, call `clear_sentry_user()` after `call_next` to clean up for the next request.
