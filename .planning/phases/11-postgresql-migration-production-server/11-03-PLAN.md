  <action>
Create a Python migration script at `ra-tracker/scripts/migrate_sqlite_to_pg.py` that:

1. **Purpose**: Migrate all data from SQLite database to PostgreSQL. This is more portable than pgloader (which has poor Windows support) and gives full control over type conversions.

2. **Usage**:
   ```bash
   python scripts/migrate_sqlite_to_pg.py --sqlite data/ra_tracker.db --pg-url postgresql://user:pass@host:5432/dbname
   ```

3. **Implementation**:
   - Accept `--sqlite` (path to SQLite DB) and `--pg-url` (PostgreSQL connection string) arguments via argparse
   - Accept optional `--drop-existing` flag to DROP and recreate tables in PostgreSQL
   - Connect to SQLite with sqlite3
   - Connect to PostgreSQL with psycopg2
   - For each table in migration order (respecting foreign keys):
     a. Read all rows from SQLite
     b. Convert types: INTEGER booleans (0/1) → Python bool for BOOLEAN columns
     c. INSERT into PostgreSQL using %s placeholders
     d. Log row count per table
   - After all inserts, run sequence resets (from reset_sequences.sql content embedded in script)
   - Wrap in transaction: commit on success, rollback on any error

4. **Table migration order** (FK-safe):
   ```
   1. users (no FKs)
   2. rules (references users but user_id is nullable)
   3. events (no FKs to other tables)
   4. event_artists (references events)
   5. event_promoters (references events)
   6. event_rules (references events, rules)
   7. notifications (references events, rules, users)
   8. sessions (references users)
   9. telegram_link_codes (references users)
   10. audit_logs (references users loosely)
   ```

5. **Boolean column mapping** (columns that need int→bool conversion):
   - rules.is_active
   - events.is_ticketed, is_festival, is_multi_day
   - users.is_admin, email_verified, telegram_enabled, email_enabled

6. **Schema creation**: Before migrating data, execute the PG_SCHEMA DDL from database.py to create tables. Import it:
   ```python
   # Add ra-tracker to path for imports
   sys.path.insert(0, os.path.dirname(os.path.dirname(os.path.abspath(__file__))))
   from ra_tracker.database import PG_SCHEMA
   ```
   Or embed the schema directly to keep the script self-contained. Embedding is safer for a one-time migration script.

7. **Sequence reset**: After data migration, reset all SERIAL sequences:
   ```sql
   SELECT setval('rules_id_seq', COALESCE((SELECT MAX(id) FROM rules), 1));
   SELECT setval('users_id_seq', COALESCE((SELECT MAX(id) FROM users), 1));
   SELECT setval('notifications_id_seq', COALESCE((SELECT MAX(id) FROM notifications), 1));
   SELECT setval('audit_logs_id_seq', COALESCE((SELECT MAX(id) FROM audit_logs), 1));
   ```
   Note: events, event_artists, event_promoters, event_rules, sessions, telegram_link_codes do NOT use SERIAL (their PKs are external IDs or text).

8. **Error handling**: Try/except around each table migration with clear error messages. If one table fails, report which table and row caused the error. Use `--verbose` flag for detailed logging.

9. **Dry-run mode**: Add `--dry-run` flag that shows what would be migrated (row counts per table) without actually writing.

10. **Script header**:
    ```python
    #!/usr/bin/env python3
    """Migrate RA Tracker data from SQLite to PostgreSQL.

    Usage:
        python scripts/migrate_sqlite_to_pg.py --sqlite data/ra_tracker.db --pg-url postgresql://user:pass@host/db

    This is a one-time migration script for transitioning from SQLite to PostgreSQL.
    Run reset_sequences.sql after migration if running separately.
    """
    ```
