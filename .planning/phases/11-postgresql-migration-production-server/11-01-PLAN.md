  <name>Task 2: Convert all SQL queries from SQLite to PostgreSQL-compatible syntax</name>
  <files>

  <action>
Convert ALL 93 SQL queries in database.py from SQLite `?` placeholders to PostgreSQL `%s` placeholders. This is a mechanical but critical conversion that must cover every query in the file.

**Strategy**: Since the application must support BOTH SQLite and PostgreSQL, use a parameterized placeholder approach:

1. **Add a placeholder property**: Add `@property` to Database class:
   ```python
   @property
   def ph(self) -> str:
       """Return the parameter placeholder for the current database backend."""
       return "%s" if self._use_postgres else "?"
   ```

2. **Convert all queries to use self.ph**: For every SQL query string that contains `?`, replace with f-string using `{self.ph}`. Examples:
   - `conn.execute("SELECT * FROM users WHERE email = ?", (email,))`
   - becomes: `conn.execute(f"SELECT * FROM users WHERE email = {self.ph}", (email,))`

3. **Convert boolean values**: Search for all places where integer 0 or 1 is used as a boolean parameter. In PostgreSQL, pass Python `True`/`False` instead. Key locations:
   - `create_user`: `is_first` is already a Python bool - OK
   - `set_email_verified`: `verified` param is already bool - OK
   - `set_user_telegram_enabled`: `enabled` param is already bool - OK
   - `set_user_email_enabled`: `enabled` param is already bool - OK
   - `get_unverified_user_by_email`: Uses `email_verified = 0` in WHERE clause. For PostgreSQL, this needs to be `email_verified = FALSE`. Use `{self._false_val}` helper.
   - Any other `= 0` or `= 1` in boolean column WHERE clauses

4. **Add boolean value helpers**:
   ```python
   @property
   def _true_val(self):
       return "TRUE" if self._use_postgres else "1"

   @property
   def _false_val(self):
       return "FALSE" if self._use_postgres else "0"
   ```

5. **Handle lastrowid**: PostgreSQL does not support `cursor.lastrowid` the same way. For INSERT statements that need the new ID, add `RETURNING id` clause for PostgreSQL:
   - `create_user`: After INSERT, use `RETURNING id` for PostgreSQL, `cursor.lastrowid` for SQLite
   - `add_rule`: Same pattern
   - `add_audit_log`: Same pattern
   - Any other INSERT that uses `cursor.lastrowid`

6. **Handle executescript**: SQLite's `conn.executescript(SCHEMA)` does not exist in psycopg2. For PostgreSQL path in init_schema, execute each CREATE TABLE statement individually via `conn.execute()`.

7. **Handle PRAGMA**: Remove `PRAGMA foreign_keys = ON` from PostgreSQL path (already handled in get_connection dual-mode).

8. **Handle datetime parsing**: PostgreSQL returns datetime objects natively. SQLite returns strings that need `datetime.fromisoformat()`. Add conditional parsing:
   - For User, Session, Rule objects: if self._use_postgres, the row values are already datetime objects
   - For SQLite: keep existing fromisoformat parsing
   - Add a helper: `_parse_datetime(self, val)` that returns val directly if it's already a datetime, or calls fromisoformat if it's a string, or returns None.

9. **Handle SELECT EXISTS**: SQLite `SELECT EXISTS(SELECT 1 FROM users LIMIT 1)` works in PostgreSQL too - no change needed.

10. **IMPORTANT edge cases to verify**:
    - `has_event_notification`: Uses `?` placeholder - convert
    - `add_event_rule_mapping`: Uses `?` - convert
    - `clear_all_events`: No params - no change needed
    - All `get_*` methods: Convert `?` to `{self.ph}`
    - All `add_*` methods: Convert `?` to `{self.ph}`, handle RETURNING id
    - All `update_*` methods: Convert `?` to `{self.ph}`
    - All `delete_*` methods: Convert `?` to `{self.ph}`
    - `get_upcoming_events_for_user`: Complex query with multiple `?` - convert all
    - `get_audit_logs`: Has dynamic WHERE clause building with `?` - convert all

Go through the ENTIRE file methodically from top to bottom. Do not skip any query. The file is 1780 lines - every `?` in a SQL string must become `{self.ph}`.
