# Milestone v2.0: Multi-User Support

**Status:** SHIPPED 2026-02-01
**Phases:** 1-4
**Total Plans:** 14

## Overview

Add multi-user support to RA Tracker with authentication, per-user data isolation, and configurable notification channels (Telegram and/or Email).

## Phases

### Phase 1: Database Schema

**Goal:** Add users table and establish foreign key relationships for multi-tenancy
**Depends on:** None
**Plans:** 1 plan

Plans:
- [x] 01-01-PLAN.md - Add users table, User dataclass, user_id migrations, and user CRUD operations

**Details:**
- Users table with id, email, password_hash, display_name, is_admin, telegram_chat_id, created_at
- Rules table extended with user_id foreign key
- Notifications table extended with user_id foreign key
- Argon2id password hashing via argon2-cffi
- First user becomes admin and receives legacy data
- Anonymous mode until first user registers

---

### Phase 2: Authentication

**Goal:** Implement user registration, login, logout with secure password hashing and session management. Migrate UI to Tailwind CSS with mobile-first responsive design.
**Depends on:** Phase 1
**Plans:** 5 plans in 3 waves

Plans:
- [x] 02-01-PLAN.md - Session infrastructure (database table, CRUD, config, auth module) [Wave 1]
- [x] 02-02-PLAN.md - Authentication routes (register, login, logout, privacy) and templates [Wave 2]
- [x] 02-03-PLAN.md - Tailwind CSS migration and mobile navigation in base.html [Wave 2]
- [x] 02-04-PLAN.md - Route protection and mobile artist tracking UI [Wave 3]
- [x] 02-05-PLAN.md - Mobile responsiveness for existing templates and verification [Wave 3]

**Details:**
- Sessions table with secure random tokens
- 30-day configurable session timeout
- httponly, secure, samesite=lax cookies
- Tailwind CSS v4 via CDN
- Hamburger menu for mobile navigation
- 44px touch targets (WCAG AAA)
- Privacy Policy page with explicit consent checkbox

---

### Phase 3: Multi-Tenant Access

**Goal:** Scope data access to logged-in user and protect routes
**Depends on:** Phase 2
**Plans:** 3 plans in 2 waves

Plans:
- [x] 03-01-PLAN.md - User-scoped database methods (user_id parameters, ownership checks) [Wave 1]
- [x] 03-02-PLAN.md - User-scoped routes (dashboard, rules page, rule mutations) [Wave 2]
- [x] 03-03-PLAN.md - Admin routes and templates (require_admin, /admin/rules, /admin/users) [Wave 2]

**Details:**
- All routes except login/register require authentication
- User only sees their own rules and notifications
- Events remain shared globally (single cache)
- Ownership verification on all mutations (returns 404 to prevent IDOR)
- Admin can view all users' rules (read-only)

---

### Phase 4: User Notifications

**Goal:** Allow each user to configure notification channels (Telegram and/or Email) with independent on/off toggles
**Depends on:** Phase 3
**Plans:** 5 plans in 3 waves

Plans:
- [x] 04-01-PLAN.md - Database schema extensions and config (notification preferences, link codes, EmailConfig) [Wave 1]
- [x] 04-02-PLAN.md - Telegram bot service with /link, /stop, /start command handlers [Wave 2]
- [x] 04-03-PLAN.md - Email sender service with SMTP and signed unsubscribe tokens [Wave 2]
- [x] 04-04-PLAN.md - Settings page UI with notification preferences and link code generation [Wave 3]
- [x] 04-05-PLAN.md - Per-user notification dispatch in scheduler [Wave 3]

**Details:**
- telegram_enabled and email_enabled columns per user
- Telegram bot responds to /link, /stop, /start commands
- Email notifications with one-click unsubscribe (no login required)
- Settings page with iOS-style toggle switches
- Per-user notification dispatch in scheduler

---

## Milestone Summary

**Key Decisions:**
- Argon2id for password hashing (OWASP 2025 recommended)
- First user becomes admin (simple bootstrap pattern)
- Tailwind v4 via CDN (no build step)
- 44px touch targets for mobile (WCAG AAA)
- Ownership verification returns 404 (never reveal if rule exists for another user)
- Email opt-out model (enabled by default, unsubscribe without login)

**Issues Resolved:**
- Global singleton pattern adapted for multi-user context
- Legacy data migration (assigned to first user on registration)
- Mobile responsiveness across all templates

**Issues Deferred:**
- Password reset via email (requires email infrastructure)
- CSRF protection on forms (deferred to security hardening milestone)
- Rate limiting on login attempts

**Technical Debt Incurred:**
- None significant - all phases completed cleanly
