<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Create hosting account and configure production environment</name>
  <files></files>
  <action>
    The user must complete these steps before Claude can deploy. These are actions that require human interaction with external dashboards and cannot be automated.

    Steps for the user:

    1. Create Render account (if not already done):
       - Go to https://render.com and sign up (GitHub OAuth recommended for auto-deploy)
       - Add billing information (Starter plan for PostgreSQL requires paid account)

    2. Create services from render.yaml:
       - Option A (Blueprint): Go to Render Dashboard -> New -> Blueprint. Connect your GitHub repo (ra-tips). Render reads render.yaml and creates all services + database automatically.
       - Option B (Manual): Create each service individually:
         a. New -> PostgreSQL -> Name: rave-tracker-db, Plan: Starter ($7/mo)
         b. New -> Web Service -> Connect repo, Name: rave-tracker-web, Root: ra-tracker, Docker
         c. New -> Background Worker -> Connect repo, Name: rave-tracker-scheduler, Root: ra-tracker, Docker

    3. Set secret environment variables:
       For BOTH the web service AND the scheduler service, set these in Render Dashboard -> Service -> Environment:
       - SECRET_KEY = Generate with: python -c "import secrets; print(secrets.token_urlsafe(32))" (Use the SAME value for both services)
       - BASE_URL = Your Render web service URL (e.g., https://rave-tracker-web.onrender.com) (Only needed on web service)
       - BREVO_SMTP_USERNAME = Your Brevo SMTP username (from Brevo Dashboard -> SMTP and API)
       - BREVO_SMTP_PASSWORD = Your Brevo SMTP API key (from Brevo Dashboard -> SMTP and API)
       - TELEGRAM_BOT_TOKEN = Your existing Telegram bot token (from @BotFather, same as local)
       Note: DATABASE_URL is auto-configured by render.yaml's fromDatabase directive. Do NOT set it manually.

    4. Note the PostgreSQL external connection URL:
       - Go to Render Dashboard -> rave-tracker-db -> Info -> External Database URL
       - Copy this URL -- needed for the SQLite data migration in Task 2
       - Format: postgresql://user:password@host/database

    5. Verify initial deployment:
       - Render should auto-build and deploy after connecting the repo
       - Wait for both services to show "Live" status in dashboard
       - The web service should respond at its .onrender.com URL
       - If build fails, check Render logs for errors (likely missing env vars or Dockerfile issues)
  </action>

<task type="auto">
  <name>Task 2: Run data migration and verify production deployment</name>
  <files></files>
  <action>
    After the user provides the Render web service URL and PostgreSQL external URL, perform these verification and migration steps:

    **Step 1: Verify health endpoint:**
    ```bash
    curl -s https://[RENDER_URL]/health
    ```
    Expected: {"status": "healthy", "database": "connected"}
    If database shows "disconnected", the DATABASE_URL env var is not set correctly.

    **Step 2: Run SQLite-to-PostgreSQL data migration:**
    The user's existing SQLite database needs to be migrated to the production PostgreSQL.

    ```bash
    cd ra-tracker
    python scripts/migrate_sqlite_to_pg.py \
      --sqlite data/ra_tracker.db \
      --pg-url "[EXTERNAL_PG_URL]" \
      --dry-run --verbose
    ```

    Review dry run output. If it looks correct:

    ```bash
    python scripts/migrate_sqlite_to_pg.py \
      --sqlite data/ra_tracker.db \
      --pg-url "[EXTERNAL_PG_URL]" \
      --verbose
    ```

    Expected: All tables migrated with row counts matching, sequences reset.

    **Step 3: Verify production data:**
    ```bash
    curl -s https://[RENDER_URL]/health
    ```
    Should still show healthy after migration.

    Then verify the login page loads:
    ```bash
    curl -s -o NUL -w "%%{http_code}" https://[RENDER_URL]/login
    ```
    Expected: 200

    **Step 4: Verify auto-deploy pipeline:**
    Check that Render is configured for automatic deploys from the main branch:
    - The Blueprint or manual setup should have connected the GitHub repo
    - Render Dashboard -> Web Service -> Settings -> Build and Deploy should show Auto-Deploy: Yes
    - If not enabled, the user should enable it in the dashboard

    **Step 5: Verify scheduler is running:**
    Check Render Dashboard -> rave-tracker-scheduler -> Logs for scheduler startup messages.
    The scheduler should show "Starting scheduler process..." or similar APScheduler initialization.

    **Step 6: Test scraper from cloud (critical -- IP blocking risk):**
    After scheduler has had time to run at least one cycle, check:
    - Render Dashboard -> rave-tracker-scheduler -> Logs for scraper execution
    - Look for 403 errors (IP blocking) or successful event fetching
    - If scraper fails with 403/429, note this for Phase 13 (Scraper Resilience)

    If the health check fails or migration has errors, debug based on Render logs and fix configuration issues before proceeding.
  </action>
  <verify>
    1. curl -s https://[RENDER_URL]/health returns {"status": "healthy", "database": "connected"}
    2. Migration script output shows all tables migrated with matching row counts
    3. curl -s -o NUL -w "%%{http_code}" https://[RENDER_URL]/login returns 200
    4. Render dashboard shows both web and scheduler services as "Live"
    5. Auto-deploy is enabled for the main branch
  </verify>
  <done>
    Application is live on Render. PostgreSQL database contains migrated data from SQLite. Health endpoint reports healthy. Login page loads. Scheduler is running. Auto-deploy is configured.
